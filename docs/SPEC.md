# Keinage Specification

最終更新: 2026-04-26

## 1. 目的と対象

Keinage は、管理画面から登録したコンテンツを表示画面へリアルタイム反映するデジタルサイネージ Web アプリです。

本書は、利用者視点の仕様を整理するためのドキュメントです。実装方針や内部構造は `docs/DESIGN.md`、HTTP エンドポイントは `docs/API.md` を参照してください。

## 2. ユーザアカウント
### 2.1 Owner
各ボードおよび管理者画面は1人のOwner Userに紐づきます。
同じボード、管理者画面を複数のOwnerで共有することはできません。

1人のOwnerに対して複数のShared Userを作成することができ、ボードの編集を共有できます。

これらは`attribute`という情報で管理され、`owner`と`shared`の二値です。

## 2.2 ロール

Keinage のUserには 2 種類のロールがあります。

| ロール | 用途 | 主な権限 |
| --- | --- | --- |
| `admin` | Ownerおよびシステム管理者 | ユーザー管理、全体設定、ボード編集、メディア管理、認証期限設定 |
| `general` | 一般運用者 | ボード編集、個人設定変更、PIN/メール/パスワード変更 |

権限差分の要点は以下です。

| 機能 | admin | general |
| --- | --- | --- |
| ボード作成・編集・削除 | ✓ | ✓ |
| 表示テーマ変更 | ✓ | ✓ |
| 自分のユーザーID変更 | ✓ | ✓ |
| 自分のメールアドレス変更 | ✓ | ✓ |
| 自分のパスワード変更 | ✓ | ✓ |
| 自分の PIN 設定・変更 | ✓ | ✓ |
| 天気地域設定 | ✓ | - |
| 画像リサイズ設定 | ✓ | - |
| ディスク上メディア管理 | ✓ | - |
| フル認証期限設定 | ✓ | - |
| ユーザー管理ページ | ✓ | - |

最後の Owner User は削除・一般権限への降格ができません。

## 3. 認証仕様

### 3.1 Owner サインアップ

Owner アカウントの登録は `/signup` から開始します。

1. `/signup` でユーザーID、メールアドレス、電話番号を登録する
2. 重複がなければ `/signingup` へ遷移する
3. SMTP が設定されていれば登録用 URL をメール送信する
4. SMTP 未設定時はメール送信を行わず、`/signingup` 画面上に登録リンクを表示する
5. `/signup/<token>` でパスワードを設定する
6. 続けて `/pin/setup` で 6 桁 PIN を設定する
7. PIN 設定後にセッションを発行し、そのままダッシュボードへ入る

補足:

- `signup` トークンの有効期限は 10 分です。
- `/signingup` は直リンクでは利用できず、仮登録時に設定される Cookie が必要です。
- `/signingup` の再送・再発行を行うと、それ以前の登録リンクは無効になります。

### 3.2 通常ログイン

通常ログインには 2 つの経路があります。

| 画面 | 入力 | 用途 |
| --- | --- | --- |
| `/pin/login` | メールアドレスまたはユーザーID + パスワード | フル認証 |
| `/pin` | 6 桁 PIN | 直近にフル認証済みのユーザーのクイック再認証 |

フル認証に成功すると `auth-session` Cookie が発行され、最後に認証したユーザーIDが `last-user-id` Cookie に保存されます。PIN 画面はこの `last-user-id` を優先して対象ユーザーを決定します。

### 3.3 フル認証期限

パスワード認証の有効期限は管理者設定で変更できます。

| 項目 | 値 |
| --- | --- |
| 既定値 | 30 日 |
| 上限 | 365 日 |
| セッション Cookie の有効期限 | 24 時間 |

フル認証期限が切れた場合、PIN が合っていてもダッシュボードへは入れず、再度 `/pin/login` でのパスワード認証が必要です。

### 3.4 PIN の扱い

- PIN はユーザーごとに保持します。
- PIN 未設定ユーザーは PIN ログインを利用できません。
- PIN は 6 桁の数字のみです。
- PIN ハッシュはメモリハード KDF で保存します。
- 旧形式の PIN ハッシュが残っている場合は、PIN 検証成功時または PIN 再設定時に新形式へ移行します。
- PIN の新規設定、変更、メールアドレス変更、ユーザーID変更はログイン済みセッション内で行います。

### 3.5 失敗回数制限

- パスワード認証と PIN 認証は、同じ `pin_attempts` テーブルで失敗回数を記録します。
- 同一 IP から 24 時間以内に 5 回失敗すると、一時的に認証をブロックします。

### 3.6 PIN リセット

`/pin/forgot` から PIN リセットを開始できます。

- 対象はメールアドレス一致ユーザーです。
- リセットトークンの有効期限は 30 分です。
- SMTP が設定されていればメール送信します。
- SMTP 未設定時は、画面でリセット URL を扱えるフォールバック動作があります。

## 4. 主な画面

### 4.1 表示画面

- `/<boardId>` でボードを表示します。
- ボードが無効 (`isActive = false`) の場合は公開対象になりません。
- 表示画面は SSE を購読し、ボード・メディア・メッセージの変更を自動反映します。

### 4.2 管理画面

主なページは以下です。

| パス | 内容 |
| --- | --- |
| `/boards` | ボード一覧 |
| `/boards/new` | ボード新規作成 |
| `/boards/[boardId]` | ボード編集 |
| `/media` | アップロード済みメディアの一覧・削除 |
| `/settings` | ユーザー設定、管理設定 |
| `/users` | ユーザー管理 (`admin` のみ) |

認証前のサインアップ導線として、`/signup`、`/signingup`、`/signup/[token]`、`/pin/setup` があります。

モバイルではサイドバーがハンバーガーメニュー化され、狭い画面でも操作可能です。

### 4.3 呼び出し画面

`call-number` テンプレート専用に `/call/<boardId>` を提供します。

- ボード設定に保存された 6 桁パスコードが必要です。
- 管理画面では呼び出し URL とパスコード込み QR コードを確認できます。
- 呼び出し画面は、ボードに紐づくアクティブなメッセージを呼び出し番号キューとして扱います。

## 5. ボードテンプレート

現在のテンプレートは次の 5 種類です。

| ID | 表示名 | 用途 |
| --- | --- | --- |
| `simple` | シンプル掲示板 | スライドショー + ティッカー |
| `photo-clock` | フォトクロック | 写真スライドショー + 日時オーバーレイ |
| `retro` | レトロ掲示板 | ドットマトリクス風デザイン |
| `message` | メッセージ掲示板 | 外部メッセージのリアルタイム表示 |
| `call-number` | 呼び出し番号 | 窓口・飲食店向け番号表示 |

各ボードは、テンプレート固有設定に加えて、色、フォント、スライド間隔などを持ちます。

## 6. コンテンツ仕様

### 6.1 ボード

- ボードは UUID を主キーに持ちます。
- `isActive` により公開可否を切り替えられます。
- `config` はテンプレート固有設定を JSON で保持します。

### 6.2 メディア

アップロード仕様は以下です。

| 項目 | 内容 |
| --- | --- |
| 対応画像 | JPEG, PNG, WebP, GIF |
| 対応動画 | MP4, WebM |
| 最大ファイルサイズ | 50 MB |
| 保存先 | `uploads/` / 必要に応じて S3 互換ストレージ |
| サムネイル保存先 | `uploads/thumbs/` / 必要に応じて S3 互換ストレージの `thumbs/` プレフィックス |

画像はアップロード時に以下の処理を行います。

- 長辺を既定 3840px にリサイズする
- 600px 長辺のサムネイルを生成する
- GIF はアニメーション保持のため元画像をリサイズせず、サムネイルのみ静止画 JPEG 化する

ボード設定の `slideInterval` は画像の既定表示時間として扱います。既定値のまま残っている既存画像については、ボード側のスライド間隔変更時に同期されます。

### 6.3 メッセージ

- メッセージはボード単位で管理します。
- `priority` で優先度を持てます。
- `expiresAt` を設定したメッセージは期限後に無効扱いになります。
- `message` テンプレートや `call-number` テンプレートでは、メッセージ内容が主要コンテンツになります。

### 6.4 天気・時計

- 天気表示は `weather.tsukumijima.net` のデータを利用します。
- 対象地域は全体設定 (`weatherCityId`) で管理します。
- 天気取得結果は 30 分キャッシュされます。
- 各テンプレート設定で時計・天気表示を切り替える構成を取ります。

## 7. リアルタイム更新

ダッシュボードまたは API からの変更後、対象ボードへ SSE イベントを発行します。

| イベント | 発生契機 |
| --- | --- |
| `board-updated` | ボード更新・削除 |
| `media-updated` | メディア追加・並び替え・削除 |
| `message-updated` | メッセージ追加・更新・削除 |

表示画面は `/api/sse/<boardId>` を購読し、イベント受信後に対象データを再取得します。

## 8. 設定項目

### 8.1 ユーザー単位

- 表示テーマ (`system` / `light` / `dark`)
- ユーザーID
- メールアドレス
- パスワード
- PIN

### 8.2 システム共通 (`admin` のみ)

- 天気地域 (`weatherCityId`)
- 画像リサイズ上限 (`imageMaxLongEdge`)
- フル認証期限 (`authExpireDays`)
- アプリの現行バージョンと最新版確認
- ダッシュボード URL / 呼び出し画面 URL の QR コード参照

## 9. 運用上の前提

- 本アプリは同一インスタンス上での運用を前提にしています。
- メディアファイルは既定でローカルディスクに保存します。必要に応じて S3 互換のあるストレージサービスを使えます。
- リアルタイム通知と天気キャッシュはメモリ上で保持するため、マルチインスタンス同期は考慮していません。
